# AGENTS.md

## 목적

이 저장소는 하나의 Discord 봇 런타임에서 `liar`, `mafia`, 이후 확장 가능한 추리/파티 게임을 운영하는 장기 프로젝트다.

- 현재 제품 축은 `liar` + `mafia`
- 단기 PMF 축은 `liar`
- 장기 플랫폼 축은 `공통 전적/운영/결제/리더보드 계층`

에이전트는 매 작업에서 다음 세 가지를 동시에 지켜야 한다.

1. 현재 게임 규칙과 제품 범위를 임의로 흐리지 않는다.
2. 장기 프로젝트에서 맥락이 끊기지 않도록 `@memory/` 를 유지한다.
3. 큰 작업은 `test-driven prompting` + `divide & conquer` 방식으로 쪼개서 진행한다.

## 제품 구조

### `liar`

- Discord 채널 기반 게임
- 핵심 파일: `liar/src/**`
- 규칙 기준 문서: `liar/RULE.md`
- 운영 설명: `liar/ABOUT.md`
- 확장 계획: `liar/PLAN.md`

### `mafia`

- Discord 로비 + 웹 대시보드 기반 게임
- 핵심 파일: `mafia/src/**`
- 규칙 기준 문서: `mafia/docs/RULE.md`
- 운영 설명: `mafia/ABOUT.md`
- 사업/플랫폼 참고: `mafia/docs/business_plan.md`, 루트 `business_plan.md`

### 공통

- 루트 `README.md`: 설치/빌드/실행/운영 진입점
- 루트 `business_plan.md`: `liar` 중심 멀티게임 사업 전략
- 공통 실행 런타임: 루트 `package.json`, `scripts/run-build.mjs`
- shared business DB: `mafia/src/db/**`

## 시즌4 마피아 범위 고정 규칙

마피아 구현을 다룰 때는 아래 범위를 벗어나지 않는다.

- `8인 이하`
- `일반/클래식`
- `예언자, 판사, 교주팀 제외`
- `듀얼 고유능력 제외`
- `룰방 로컬 규칙 제외`

## 시즌4 마피아 소스 우선순위

규칙을 확인할 때는 반드시 아래 순서를 따른다.

1. `시즌4 업데이트 문서`
2. `당시와 충돌하지 않는 기본 규칙 문서`
3. `당시와 충돌하지 않는 직업 문서`
4. `해당 직업의 출시 공지`
5. 그래도 비면 `미확정`

후대 문서 표현이나 효과를 시즌4 규칙처럼 섞어 쓰지 않는다.

## 시즌4 내부 시점 구분

- `S4-초기`: `2016-10-10 16:00` 이후, `2017-01-13 15:59` 이전
- `S4-밸런스`: `2017-01-13 16:00` 이후

참고:

- `성직자` 추가: `2016-10-10 16:00`
- `예언자`, `판사` 추가: `2017-03-24 16:00`
- 이 저장소는 `2017-03-24` 이후에도 `예언자`, `판사`를 범위 밖으로 유지한다.

## 현재 허용 직업

- 마피아팀: 마피아, 스파이, 짐승인간, 마담
- 시민팀: 경찰, 의사, 군인, 정치인, 영매, 연인, 건달, 기자, 탐정, 도굴꾼, 테러리스트, 성직자
- 이 저장소에서 `탐정`은 마피아42의 `사립탐정`을 뜻한다.

## 규칙 작업 원칙

- 게임별 단일 기준 문서는 각 모듈의 `RULE.md` 다.
- `mafia/docs/RULE.md` 에는 반드시 `확정`, `복원`, `미확정`을 구분한다.
- 시즌4 문서와 후대 직업 문서가 충돌하면 시즌4 문서를 우선한다.
- 후대 직업 문서는 시즌4 문서가 비워 둔 항목만 보강하는 용도로만 쓴다.
- 애매한 상호작용을 임의 해석해서 코드에 넣지 않는다.
- 서버 구현에 필요한 전역 우선순위표가 공개 자료만으로 확정되지 않으면 문서에 그대로 남긴다.
- `liar` 도 규칙 변경 전에는 `liar/RULE.md` 와 실제 구현 상태를 함께 확인한다.

## Discord 구현 가이드

- Discord는 전달 수단일 뿐이며, 규칙 자체는 해당 게임 기준 문서를 우선한다.
- 비공개 정보는 DM, ephemeral, 웹 대시보드 등 적절한 비공개 인터페이스로 전달한다.
- 공개 정보만 공개 채널에 노출한다.
- 플레이어용 문구와 명령어는 한국어 기준으로 설계한다.
- `liar` 는 장기적으로 `Message Content Intent` 의존을 줄이고 `slash/button/select` 비중을 높인다.
- `mafia` 는 Discord 로비와 웹 대시보드 사이의 역할 분리를 유지한다.

## 다음 단계 기준

- 엔진 구현 전, 각 게임의 `RULE.md` 와 현재 테스트를 먼저 확인한다.
- `찬반 투표 시간`, `4~7인 원본 표`, `연인 배치 내부 로직`, `전역 행동 우선순위`처럼 공개 자료로 확정되지 않은 항목은 별도 플래그 없이 하드코딩하지 않는다.
- `liar` 에서는 제시어 난이도, 리매치 유도, 완료율, 방장 운영성처럼 PMF와 직결되는 요소를 먼저 다룬다.

## 메모리 운영 규칙

이 저장소의 장기 메모리는 `@memory/` 디렉토리다.

### 시작 전 필수 읽기 순서

1. `AGENTS.md`
2. `@memory/README.md`
3. `@memory/current_context.md`
4. `@memory/repo_map.md`
5. `@memory/data_models.md`
6. `@memory/milestones.md`

### 항상 유지할 메모리 파일

- `@memory/current_context.md`
  - 현재 프롬프트 진행 상황
  - 최근 완료 작업
  - 활성 제약
  - 즉시 다음 후보 작업
- `@memory/repo_map.md`
  - 폴더 구조
  - 주요 문서/모듈 위치
  - 어떤 작업을 어디서 시작해야 하는지
- `@memory/data_models.md`
  - 핵심 타입
  - 게임 상태 객체
  - 전적/DB 구조
- `@memory/milestones.md`
  - 단계별 마일스톤
  - 현재 진행 단계
  - 마일스톤별 종료 조건
- `@memory/agent_changelog.md`
  - 에이전트 지침 변경 기록

### 메모리 업데이트 규칙

- 코드 구조가 바뀌면 `@memory/repo_map.md` 를 갱신한다.
- 상태/타입/DB 구조가 바뀌면 `@memory/data_models.md` 를 갱신한다.
- 큰 작업을 끝내거나 방향이 바뀌면 `@memory/current_context.md` 를 갱신한다.
- 마일스톤 상태가 바뀌면 `@memory/milestones.md` 를 갱신한다.
- `AGENTS.md` 또는 `.clinerules` 를 바꾸면 `@memory/agent_changelog.md` 를 갱신한다.
- 메모리는 장황한 작업 로그가 아니라 `다음 에이전트가 바로 이어받을 수 있는 압축된 상태`여야 한다.

## Test-Driven Prompting 원칙

이 저장소는 가능하면 `요구사항 -> 테스트/검증 기준 -> 구현 -> 재검증` 순서로 진행한다.

### 기본 흐름

1. 프롬프트를 `행동`, `상태`, `예외`, `출력` 단위로 쪼갠다.
2. 기존 테스트가 있으면 먼저 읽고, 없으면 가장 작은 실패 검증부터 추가한다.
3. 테스트가 의미 있게 실패하는지 확인한다.
4. 최소 구현으로 통과시킨다.
5. 관련 테스트만 먼저 돌리고, 필요하면 전체 테스트로 넓힌다.
6. 변경 후 메모리를 갱신한다.

### 테스트 작성 기준

- `liar` 변경은 우선 `liar/tests/**`
- `mafia` 엔진 변경은 우선 `mafia/tests/game.test.ts` 또는 관련 서브테스트
- 전적/저장소 변경은 `mafia/tests/*stats*`, `*record*`, `dashboard-server.test.ts` 계열
- 웹 접근/권한 변경은 `mafia/tests/web-permissions.test.ts`, `dashboard-access.test.ts`

### 예외

- 문서 작업, 리서치 작업, 운영 메모 정리는 테스트보다 문서와 메모 업데이트가 우선이다.
- 테스트를 추가할 합리적 위치가 아직 없으면 먼저 테스트 대상 구조를 작게 분리한 뒤 테스트를 넣는다.

## Divide & Conquer 원칙

큰 작업은 반드시 작은 마일스톤으로 분할한다.

### 분할 기준

- 게임 단위: `liar` / `mafia` / shared DB / 공통 운영
- 계층 단위: `domain model -> orchestration/service -> discord/web presentation -> persistence`
- 위험 단위: 규칙 해석, 상태 전이, 네트워크 I/O, 전적 정산, UX 텍스트

### 각 마일스톤에 반드시 있어야 할 것

- 목표
- 건드리는 파일 범위
- 추가/수정 테스트
- 종료 조건
- 다음 마일스톤과의 연결점

### 금지

- 규칙 해석과 UI 리뉴얼과 DB 리팩터를 한 번에 섞는 작업
- 테스트 없이 대규모 상태 전이 변경
- 메모리 갱신 없는 장시간 연속 구현

## 프롬프트 유의적 버전 관리 정책

에이전트 지침 구조는 Git Tag 기반 `Prompt Semantic Versioning` 으로 관리한다.

- 형식: `v[Major].[Minor].[Patch]-[AgentName]`
- `Major`: 핵심 목적, 메모리 구조, 아키텍처 수준 변경
- `Minor`: 행동 지침 추가, 워크플로 변경, 도구 사용 정책 조정
- `Patch`: 오탈자, 톤, 작은 제약 보강

실제 태그를 만들 때는 `@memory/agent_changelog.md` 에도 기록한다.

## 작업 체크리스트

작업 전:

- 관련 `RULE.md`, `ABOUT.md`, 테스트, `@memory/*` 를 읽었는가
- 이번 작업의 마일스톤이 `@memory/milestones.md` 에 반영되어 있는가

작업 중:

- 범위를 한 단계로 유지하고 있는가
- 미확정 규칙을 코드로 확정하고 있지 않은가
- 기존 테스트와 충돌하지 않는가

작업 후:

- 관련 테스트를 돌렸는가
- 실행하지 못했다면 이유를 남겼는가
- `@memory/*` 를 갱신했는가
- 문서 진입점이 바뀌었다면 `README.md` 도 갱신했는가
