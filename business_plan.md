# business_plan.md

## 문서 목적

- 이 문서는 `discord_mafia`를 `liar` 중심의 초기 제품-시장 적합성 검증 단계에서, 장기적으로 `liar`, `mafia`, 이후 확장 게임까지 품는 `Discord 소셜 추리게임 운영 플랫폼`으로 키우기 위한 사업 계획서다.
- 기준 시점은 `2026-04-13` 이며, 공개 웹 리소스와 현재 저장소 상태를 함께 반영한다.
- 현재 최우선 범위는 `1개 서버에서 운영 안정성 달성`과 `정말 계속 하고 싶어지는 수준의 재미`다.

## 한 줄 전략

`liar`를 가장 빠르게 웃음과 긴장을 터뜨리는 진입 게임으로 만들고, `mafia`를 깊은 몰입과 누적 경쟁을 만드는 코어 게임으로 두며, 둘을 공통 기록/운영/과금 계층 위에 올려 `멀티게임 Discord 봇`이 아니라 `소셜 추리게임 운영 서비스`로 발전시킨다.

## 현재 제품 정의

### `liar`

- 역할/제시어/투표/추리로 끝나는 `10~20분`급 고회전 게임
- Discord 채널 안에서 거의 모든 흐름이 끝나야 한다
- 신규 유저가 설명 없이도 바로 붙을 수 있어야 한다
- 한 판이 끝나면 `한 판 더`가 자연스럽게 나와야 한다

### `mafia`

- 더 긴 몰입, 역할 조합, 전적, 시즌, 대시보드가 어울리는 심화 게임
- `liar`에서 유입된 그룹이 더 진한 경험을 원할 때 넘어가는 상위 코어 게임

### 플랫폼 공통

- 공통 계정/길드/전적/리더보드/운영 로그/권한/결제 계층을 공유한다
- 게임별 UI 매체는 다를 수 있다
- 외부 브랜딩은 특정 원작 복제보다 `Discord 소셜 추리게임 플랫폼`에 가깝게 잡는다

## 리서치에서 확인한 핵심 사실

### 장르 쪽

- Oink Games `A Fake Artist Goes to New York` 는 `5~10인`, `20분` 구조이며, `카테고리 힌트 제공`, `짧은 개인 행동`, `정체 노출 후 마지막 정답 추리`가 핵심 긴장 장치다.
- Oink Games `INSIDER` 는 `4~8인`, `15분` 구조이며, `정답 찾기`와 `내부자 찾기` 두 층의 목표를 겹쳐 짧은 시간 안에 밀도를 높인다.
- `Spyfall` 류 구조는 `모두가 같은 정보 일부를 공유`하고 `소수 역할만 핵심 정보가 비는` 방식이 반복 플레이에 강하다.
- 한국어 웹 라이어게임 서비스 예시도 `설명 30초`, `투표 3분`, `변론 1분`처럼 강한 타이머 기반 흐름을 쓴다.

### Discord 쪽

- Discord 공식 회사 페이지 기준 `Q4 2025` 에 `90M+` 일일 활성 이용자가 있고, `90%+` 는 비디오 게임을 하며, PC 이용자 중 `약 40%` 는 `1시간 안에 게임을 시작`한다.
- Discord 공식 도움말 기준 App Directory 검색 결과는 `관련성`과 `사용 기반 인기`에 따라 정렬된다. 즉, `설치`보다 `반복 사용`이 더 중요하다.
- App Directory 탐색은 공식 도움말 기준 `desktop/browser` 에서 이뤄진다. 설치 이후 플레이 경험은 모바일도 고려해야 하지만, 초기 발견 경로는 데스크톱 중심으로 봐야 한다.
- Discord 공식 케이스 스터디에서 Playroom의 `Death by AI` 는 출시 후 수 주 내 `거의 700만 사용자`에 도달했고, 첫 주 플레이 세션의 `70%+` 가 `친구 3명 이상`과 함께였으며, `30%+ Day 7 engagement` 를 여러 주 유지했다.
- 같은 케이스 스터디에서 Discord는 `guild/global stats`, `비동기 재참여`, `업데이트 재호출`, `호스트의 문제 유저 통제`를 반복 사용의 핵심으로 제시한다.

### 운영/기술 쪽

- Discord 공식 문서 기준 `MESSAGE_CONTENT` 는 검증된 앱에서 특권 인텐트다. 따라서 장기적으로 핵심 조작을 `slash command`, `button`, `select`, `context menu` 중심으로 옮기는 편이 안전하다.
- Discord 공식 문서 기준 일반 HTTP API는 봇당 `초당 50 요청` 글로벌 제한이 있고, 라우트별 제한은 고정값 하드코딩이 아니라 헤더 파싱으로 대응해야 한다.
- Discord Premium Apps 문서 기준 `2026-04-13` 현재 미국, 유럽연합, 영국 외 개발자에게는 기본 제공 수익화가 열려 있지 않다. 한국 기반 팀이면 당분간 `외부 웹 결제` 또는 `후원형 결제`가 현실적이다.
- App Directory 정책상 `Privacy Policy`, `Terms of Service`, `IP 위반이 없는 명칭/설명/명령어`, 연령 제한성 콘텐츠 회피가 필요하다.

## 제품 가설

### 핵심 가설 1

`liar` 는 `짧은 설명`, `강한 정보 비대칭`, `마지막 역전 찬스`, `사람끼리 웃으면서 싸울 수 있는 판 구조`가 맞물릴 때 가장 빠르게 반복 플레이를 만든다.

### 핵심 가설 2

`mafia` 는 `더 긴 세션`, `역할 다양성`, `누적 전적`, `길드 내 경쟁`이 붙을 때 가치가 커진다.

### 핵심 가설 3

초기 PMF는 `서버 수`보다 `한 서버 안에서의 재방문 밀도`로 봐야 한다. 즉, 첫 성공 기준은 `한 서버에서 일주일 내 여러 번 자발적 재개최가 일어나는가`다.

## 왜 사람들이 계속 하게 되는가

반복 플레이는 아래 다섯 축이 동시에 맞아야 나온다.

### 1. 시작 마찰이 낮아야 한다

- 방 생성부터 시작까지 `1분 안팎`
- 설명이 없어도 `참가`, `제시어 확인`, `말하기`, `투표`만 이해하면 된다
- 한 명이 규칙을 외워서 설명해 주는 구조를 없앤다

### 2. 한 턴마다 긴장이 있어야 한다

- 각 플레이어의 발언이 짧고 의미 있어야 한다
- 너무 장황한 설명은 텐션을 죽인다
- `한 줄`, `짧은 타이머`, `턴 경고음`, `자동 다음 진행`이 중요하다

### 3. 공개 정보가 매 턴 누적되어야 한다

- 이전 사람의 말이 다음 사람의 말에 압박을 줘야 한다
- 관전자도 `지금 누가 이상한지` 추적할 수 있어야 한다
- 채팅 로그와 상태 메시지가 `지금 판의 서사`를 쌓아야 한다

### 4. 마지막에 폭발이 있어야 한다

- 라이어 공개
- 마지막 정답 추리
- 시민 쪽의 `너무 대놓고 말했네` 혹은 라이어 쪽의 `그걸 맞힌다고?` 같은 감정 폭발

### 5. 끝난 직후 다시 누를 이유가 있어야 한다

- 즉시 `리매치` 버튼
- 방금 판의 `결정적 한마디`, `가장 수상했던 설명`, `역전 순간` 요약
- 개인 전적, 길드 연승, 오늘의 승률 같은 누적 결과

## `극한의 재미`를 위한 설계 원칙

### 원칙 1. 죽은 시간을 없앤다

- 로비 대기, 설명 대기, 집계 대기를 최소화한다
- 호스트가 타이머를 기다리기만 하는 상황을 줄인다
- `강제 집계`, `즉시 시작`, `과반 스킵`은 유지하되 흐름을 더 빠르게 다듬는다

### 원칙 2. 발언의 가치 대비 길이를 줄인다

- `짧게 말해야 더 재미있다`
- 기본 모드는 `한 줄 설명`
- 장문의 토론은 `mafia` 쪽 가치이고 `liar` 에서는 과하면 독이다

### 원칙 3. 오디오/비주얼은 과하지 않게 강하게 쓴다

- 시작음, 막판 경고음, 결과음은 강하게
- 상시 배경음은 선택형으로 두고, 피로도를 줄인다
- 리듬은 `조용한 긴장 -> 경고 -> 폭발` 순으로 설계한다

### 원칙 4. 매 판마다 다른 느낌을 줘야 한다

- 카테고리 회전
- 희귀 제시어
- 주간 추천 팩
- 라운드 말미 `이번 판은 이런 포인트가 있었다` 식의 후처리

### 원칙 5. 웃음과 의심을 같이 만든다

- 너무 쉬운 제시어는 재미가 없다
- 너무 어려운 제시어는 억까가 된다
- `시민도 애매하게 말해야 하고 라이어도 주제만 보고 버텨야 하는` 중간 지점을 데이터로 맞춘다

### 원칙 6. 리매치를 제품의 기본 종료 화면으로 둔다

- 종료 화면의 첫 CTA는 `나가기` 가 아니라 `리매치`
- 점수판보다 `또 할 사람` 흐름이 앞에 와야 한다

- `방장 재개최율`

- `설명 단계 타임아웃 비율`

### 1차 목표 수치

- 매치 완료율 `95%+`

- 방 생성 후 시작까지 중앙값 `2분 이하`
- 판 종료 후 `10분 내 리매치율 35%+`
- 첫 플레이어의 `7일 내 재참여율 25%+`
- 방장 기준 `주 2회 이상 개최` 서버 1곳 확보

## 1개 서버 안정화 범위에서 해야 할 것

### 운영 안정성

- 서버당 동시 게임 `1개` 유지
- 상태 꼬임 시 `정리 가능한 실패`를 우선한다
- 재시작 복구가 없더라도 `깨끗한 취소 종료`는 보장한다
- 모든 단계 전환을 구조화 로그로 남긴다
- 전적 저장 실패와 게임 진행 실패를 분리한다

### UX 안정성

- `/제시어` 는 언제 눌러도 즉시 재확인 가능해야 한다
- 진행 메시지 하나만 봐도 `지금 누구 차례인지`, `무엇을 해야 하는지`, `얼마나 남았는지`가 보여야 한다
- 모바일에서 읽히는 길이로 잘라야 한다
- 음성 채널 유저와 채팅 채널 유저가 섞여도 진행을 놓치지 않아야 한다

### 사회적 안정성

- 방장은 문제 유저를 즉시 정리할 수 있어야 한다
- 게임 중 이탈, 장난성 스팸, 반복 지연 행위에 대한 운영 옵션이 필요하다
- 추리게임은 재미와 동시에 감정 소모가 있으므로 `리포트`보다 먼저 `호스트 통제`가 중요하다

## 사업 모델

### 0단계. PMF 전

- 사실상 무료 운영이 맞다
- 목표는 돈이 아니라 `정말 다시 하게 되는가` 검증이다
- 후원 링크, 테스트 서버 모집, 피드백 그룹 운영 정도만 붙인다

### 1단계. 서버 운영자 과금

초기 수익화는 `서버 단위 구독`이 가장 적합하다.

- 서버별 상세 전적 보드
- 길드 리더보드
- 이벤트 모드 예약
- 운영 로그 검색
- 커스텀 팩 관리
- 주간 리포트
- 우선 지원

### 2단계. 플레이어 과금

개인 과금은 누적 전적 가치가 생긴 뒤에만 붙인다.

- 고급 프로필
- 개인 히스토리 검색
- 시즌 회고 리포트
- 배지/칭호/프로필 꾸미기
- 개인 하이라이트 카드 저장

### 3단계. 이벤트/크리에이터 도구

- 스트리머 서버용 이벤트 모드
- 방송용 결과 오버레이
- 대회 운영 로그
- 참가자 체크인
- 시즌 이벤트 스폰서 팩

## 최종 제안

가장 좋은 경로는 `liar` 를 `한 서버에서 폭발적으로 다시 하게 되는 게임`으로 먼저 완성하고, 그 위에 `전적`, `길드 경쟁`, `운영 도구`, `mafia` 전환을 쌓는 것이다.

즉, 지금의 사업 목표는 `많은 서버에 퍼뜨리기`가 아니라 `한 서버에서 사람들의 주간 놀이 루틴이 되는 것`이다. 그 기준을 통과하면 멀티게임 플랫폼과 과금은 훨씬 자연스럽게 붙는다.
